# Zero-Knowledge Account-Password Recovery

Scope: protocol version `gvault-recovery-v1`, implemented by the GVault server,
Web client, and Android client. This protocol resets the **server account
password**. It does not reset or recover a lost vault master password.

## Security goals

- Possession of the vault master password can authorize an account-password
  reset without sending the master password, a master-password hash, or a
  reusable master-password verifier to the server.
- A server database snapshot does not contain a reusable recovery secret.
- The recovery secret is never transmitted outside its master-protected
  authenticated-encryption envelope.
- Every successful reset rotates the recovery secret. A challenge and a prior
  recovery secret cannot be replayed.
- Account existence and recovery enrollment are not disclosed by the public
  challenge endpoint.

This protocol cannot recover a forgotten master password. An attacker who has
the master password and can reach the recovery API can reset the account
password; that is the intended recovery authority.

## Version 1 primitives

| Purpose | Primitive |
| --- | --- |
| Recovery token | Fresh P-256 ECDSA private key generated by the client |
| Server verifier | X.509 SubjectPublicKeyInfo for the matching P-256 public key |
| Envelope KDF | PBKDF2-HMAC-SHA-256, 210,000 iterations, random 16-byte salt |
| Envelope AEAD | AES-256-GCM, random 12-byte nonce, 128-bit authentication tag |
| Associated data | UTF-8 `gvault-recovery-v1` |
| Challenge | Random 32-byte server nonce, bound to a random challenge id |
| Proof | ECDSA-SHA-256 signature over the canonical challenge message |

The encrypted plaintext is a fixed 256-byte payload: a two-byte big-endian
PKCS#8 length, the recovery private-key PKCS#8 bytes, and random padding for the
remaining bytes. Fixing the plaintext length prevents the public response from
revealing which client/provider created an envelope. The envelope contains only
`version`, `kdf`, `iterations`, `salt`, `nonce`, and `ciphertext`. Base64 is
standard padded base64. The public verifier is sent to
the server during registration, opt-in setup, and rotation. Private key bytes
exist only in client memory while creating a proof or a replacement envelope.

The canonical signed message is exactly these UTF-8 lines, including the two
newline separators and no trailing newline:

```text
gvault-recovery-v1
<challengeId>
<challenge>
```

Signatures are ASN.1 DER ECDSA signatures. Web converts WebCrypto's P1363
signature to DER; Android's `SHA256withECDSA` produces DER directly.

## Flows

### New account (mandatory enrollment)

1. The client validates account and master passwords locally.
2. The client generates a fresh P-256 key pair.
3. It encrypts the PKCS#8 private key under the master password and sends only
   `{ verifier, envelope }` with the normal account registration request.
4. The server validates the public verifier and envelope shape, stores them
   with the account-password scrypt hash, and returns a session.

Registration never includes the master password or private recovery token. A
new version-1 client registration is rejected if recovery material is absent;
there is no newly-created unenrolled account state.

### Existing-account explicit opt-in or rotation

Accounts created before protocol version 1 remain unenrolled until the user
chooses an authenticated and locally unlocked `Enable / rotate recovery`
action. An authenticated and locally unlocked client exposes this explicit
`Enable / rotate recovery` action. The user re-enters the current account
password. The client creates fresh recovery material and calls
`POST /api/auth/recovery/setup`. The server re-authenticates the account
password before atomically replacing any prior verifier and envelope. There is
no silent migration and no fallback for unenrolled legacy accounts.

### Forgotten account password

1. The client calls `POST /api/auth/recovery/challenge` with the normalized
   email address.
2. The server always returns HTTP 200 and the same response shape. Enrolled
   accounts receive their real envelope; unknown or unenrolled accounts receive
   a fixed-size dummy envelope. A random challenge is stored server-side with a
   short expiry and is bound to the account internally.
3. The client decrypts the envelope locally with the master password. Failure
   is reported with the same generic recovery error used for all proof failures.
4. The client signs the canonical challenge message, creates a fresh key pair
   and envelope, and calls `POST /api/auth/recovery/complete` with the
   challenge id, DER signature, new account password, and replacement public
   verifier/envelope. Neither master password nor private key is sent.
5. The server consumes the challenge before proof verification, verifies the
   signature, validates the replacement material, updates the account-password
   scrypt hash, rotates recovery material, and returns a new session.

A consumed, expired, unknown, or incorrectly signed challenge returns the same
generic `401 Recovery could not be completed` response. The previous verifier
is unusable immediately after rotation.

## Abuse controls and audit boundary

- Challenge creation and completion are separately limited by hashed email/IP
  and IP/challenge buckets. Limits use the same behavior for known and unknown
  accounts; a limited request returns `429 Recovery temporarily unavailable`.
- Challenges expire after five minutes and are single-use even when proof
  verification fails.
- Audit events contain only timestamp, event/outcome, and truncated SHA-256
  identifiers for email and remote address. Passwords, envelope plaintext,
  private keys, signatures, bearer tokens, and full email addresses are never
  logged.
- The API is intended only behind HTTPS. TLS protects public metadata,
  challenge traffic, the new account password, and the returned session in
  transit; the recovery private key still never leaves the client.

## Enumeration response contract

The public challenge response always contains:

```json
{
  "protocol": "gvault-recovery-v1",
  "challengeId": "opaque random id",
  "challenge": "opaque random nonce",
  "expiresAt": "ISO-8601 timestamp",
  "envelope": {
    "version": 1,
    "kdf": "PBKDF2-SHA256",
    "iterations": 210000,
    "salt": "base64",
    "nonce": "base64",
    "ciphertext": "base64"
  }
}
```

The response has no `exists`, `configured`, user id, or account metadata field.
Clients must use the generic failure copy and must not infer enrollment from a
local decryption failure.

## Cross-client reuse contract

Web and Android implement version 1. Browser extension, Windows, and Linux do
not yet have real account-auth flows; when they do, they must reuse the exact
primitives, envelope fields, associated data, canonical message, DER signature,
generic failures, and rotation rules above. They must not add a master-password
verifier or an operator recovery bypass.

## Residual risks

- A compromised client while the master password is in memory can use recovery.
- A weak master password permits offline guessing against the public envelope;
  the existing 12-character minimum is only a floor, not a strength guarantee.
- Availability of in-memory challenges and rate-limit buckets is tied to the
  server process. A restart invalidates active challenges and clears buckets;
  it never makes a proof reusable.
- Recovery does not revoke pre-existing bearer sessions; session expiry and
  revocation remain tracked by #483.
